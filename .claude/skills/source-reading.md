# 源码阅读方法论 v3：六层递进框架

## 核心理念

按六层框架生成**高质量、自包含、图表丰富**的分析文档：
- 文档中的 Q&A 结构激发思考，Evidence 和 Inference 清晰分离
- 每个概念附带场景 example 和结构 example，让读者一眼看懂
- 所有 `file:line (FuncName)` 引用可跳转验证，保证事实正确性

## 分级递归策略

六层框架递归应用，先对整个项目走一遍建立全局认知，再对重要模块逐个深入。

- **第一级：全局** — 每一层面向系统整体
- **第二级：模块** — 同一框架，含义下沉到模块内部

深入程度取决于模块重要性：
- 核心模块：完整六层
- 重要模块：前四层
- 辅助模块：前一两层

分级标准：
- 核心：出现在主执行路径 + 被 ≥3 个其他包 import
- 重要：被核心模块直接依赖
- 辅助：其他

## 探索顺序

1. 入口：main.go / cmd/ → 找到所有用户可触发的命令
2. 调用链：从入口沿函数调用向下追踪，建立主路径认知
3. 核心模块：主路径上调用最频繁、逻辑最复杂的包
4. 辅助模块：被核心模块引用但职责单一的包

## 文档存放位置

```
docs/source-reading/
├── 00-global.md          ← 全局六层分析
├── 01-<核心模块>.md       ← 核心模块六层分析
├── 02-<重要模块>.md       ← 重要模块前四层
└── ...
```

## 文档元数据要求

每个文档头部必须包含以下元数据：

```markdown
---
skill_version: "v3"  # 生成此文档时使用的 Skill 版本
generated_at: "YYYY-MM-DD"

  - path/to/file1.go
  - path/to/file2.go
freshness_scope: "描述哪些变更会让此文档需要更新"
---
```

## 信息来源冲突裁决

当不同来源对同一事实描述矛盾时，按以下优先级裁决：
实现代码 > 测试用例 > 代码注释 > README / 外部文档

## 六层框架定义

### Layer 1: 场景（Problem Space）

**核心问题**：为谁解决什么问题？

**输出要求**：
- 一句话描述（能对非技术人员说清楚）
- 核心用户和核心操作（谁在什么时候触发什么）
- 主场景 vs 辅助场景

**必须包含的图表**：
- **用户旅程图**：触发者 → 操作 → 系统 → 输出 → 反馈回用户

**Q&A 要求**：至少 1 个"为什么不用现有方案 X"的问题，使用 Evidence/Inference 格式

**信息来源优先级**：CLI 入口 / API 路由 > 测试用例 > README

---

### Layer 2: 概念（Domain Model）

**核心问题**：领域词汇表是什么？概念间关系如何？

**输出要求**：
- 5-7 个核心概念，分三类：核心实体 / 过程概念 / 策略概念
- 每个概念标注"作者创造的"还是"通用模式"
- 每个概念一段简洁定义（2-3 句话）
- **每个概念必须附带双层 example**：
  - **场景 example**：什么时候 / 怎么产生这个概念（如"配了 gpt-4o 和 claude-sonnet，就有 2 个 Reviewer 实例"）
  - **结构 example**：这个概念的属性是什么，每个属性给具体值，格式如：
    ```
    Reviewer {
      ID:           "reviewer-0"              // 配置文件中的 key
      Provider:     OpenAIProvider             // 由 model 字段决定的 provider 类型
      SystemPrompt: "You are a code reviewer…" // 引导审查角度的提示词
      Model:        "gpt-4o"                   // 底层模型，影响审查质量和成本
    }
    ```
    每个属性须加行内注释说明用途，让读者一眼看懂这个概念"长什么样"

**必须包含的图表**：
1. **概念关系图** — 展示核心概念之间的关系
2. **概念生命周期表** — 有状态的概念，列出其状态转换

**Q&A 要求**：至少 1 个"概念 A 和概念 B 的区别是什么"的问题，使用 Evidence/Inference 格式

**信息来源**：types 文件 > 接口定义 > 枚举/常量

---

### Layer 3: 约束（Design Boundaries）

**核心问题**：设计空间的边界在哪？

**输出要求**：
- 3-5 个关键约束，分三类：技术约束 / 业务约束 / 运行时约束
- 每个约束说明它如何影响了代码结构

**必须包含的图表**：
1. **约束-影响映射表**：约束 → 影响的设计决策

**Q&A 要求**：至少 1 个"如果没有这个约束，设计会怎样不同"的问题，使用 Evidence/Inference 格式

**信息来源**：错误处理逻辑 > 配置项 > 接口设计 > TODO/HACK 注释

---

### Layer 4: 拆分理由（Design Decisions）

**核心问题**：为什么这样拆？如果不这样拆会怎样？

**输出要求**：
- 每个主要模块/文件的拆分动机，从以下类型中识别：

| 动机 | 识别信号 | 反问 |
|------|---------|------|
| 可替换性 | 接口 + 工厂 | 如果只有一个实现，还需要接口吗？ |
| 正交性 | 两个维度独立变化 | 如果合在一起，改 A 要动 B 吗？ |
| 生命周期不同 | 一个是配置，一个是状态 | 如果合在一起，并发安全吗？ |
| 并发边界 | 各自独立更新，最后合并 | 如果共享状态，能并行吗？ |
| 演进路径 | 两套实现并存 | 旧的还在用吗？新的完全替代了吗？ |
| 失败隔离 | non-fatal 处理 | 如果这个功能挂了，主流程还能走吗？ |

**必须包含的图表**：
1. **模块依赖图** — 谁依赖谁，箭头方向 = 依赖方向
2. **职责矩阵**（模块级拆分时）— 模块内每个文件负责什么

**Q&A 要求**：每个拆分给出"如果不这样拆会怎样"的反事实分析，使用 Evidence/Inference 格式。至少一个 Q&A 须质疑现有拆分是否过度或不足，不得全部论证合理性。

---

### Layer 5: 数据流（Data Transformation）

**核心问题**：数据如何变换和流转？

**输出要求**：
- 完整数据变换链：输入格式 → 中间格式 → ... → 输出格式
- 每个变换点标注：谁做的、为什么需要这个中间格式
- 标注分叉点（一份数据被多消费者使用）和汇合点（多份数据合并）

**必须包含的图表**：
1. **数据变换管线图** — 核心数据从输入到输出的完整变换链
2. **核心类型速查表** — 列出 5-10 个最重要的类型及其关键字段

**Q&A 要求**：至少 1 个"为什么需要中间格式 X，不能直接从 A 到 C 吗"的问题，使用 Evidence/Inference 格式

**信息来源**：函数签名 > 类型转换 > 序列化/反序列化

---

### Layer 6: 执行流（Execution Path）

**核心问题**：关键路径的调用顺序、并发策略、错误处理如何？

**输出要求**：
- 主路径（happy path）的调用序列，标注文件名和关键函数
- 并发边界：哪些步骤并行、用什么原语
- 关键错误处理分支：失败后走哪条路

**必须包含的图表**：
1. **时序图** — 主路径的调用序列，标注并发边界
2. **错误处理流程图** — 关键失败点及其降级路径
3. **关键函数索引** — 列出 10-15 个最重要的函数

**Q&A 要求**：至少 1 个"如果步骤 X 失败了会怎样"的问题，使用 Evidence/Inference 格式

**信息来源**：入口函数 > 并发原语 > defer/cleanup > context 传播

## Q&A 格式规范

所有 Q&A 统一使用 Evidence/Inference 格式：

```markdown
> **Q**: [一个能检验理解深度的问题]
>
> **Evidence**: [直接来自代码的事实证据，标注 file:line (FuncName)]
>
> **Inference**: [基于证据的推断和分析，明确标注这是推断]
```

当答案完全基于代码事实时，可省略 Inference 部分。当答案完全是推断时，必须明确说明证据缺失。

## 输出格式规范

每个文档的每一层按以下结构输出：

```markdown
## Layer N: 层名

[正文内容]

[图表：该层要求的所有图表]

### Q&A

> **Q**: [问题]
>
> **Evidence**: [代码事实]
>
> **Inference**: [推断分析]
```

### Code Insights 部分

每个文档末尾附加 Code Insights：

```markdown
## Code Insights

### Insight N: [技巧名称]
**代码位置**：`file:line (FuncName)`
**是什么**：[一句话描述这个技巧]
**代码示例**：
\```go
// 从项目中摘取的关键代码片段（精简到核心逻辑）
\```
**设计意图**：[作者试图解决什么问题]
**优点**：[这个做法好在哪]
**局限性**：[这个做法的代价、风险、或更好的替代方案]
**适用场景**：[什么条件下值得采用，什么条件下不值得]
```

**Code Insights 覆盖维度**（按项目实际情况选取）：

| 维度 | 关注点 |
|------|--------|
| 设计模式实战 | 项目中用了哪些模式？为什么选这个模式而非其他？ |
| 语言惯用法 | 该语言的地道写法，非通用写法 |
| 并发技巧 | 并发原语的选择、竞态预防、同步策略 |
| 错误处理策略 | 重试、降级、non-fatal、错误链 |
| 接口设计 | 接口粒度、抽象层次、扩展性 |
| 性能考量 | 预分配、池化、批处理、懒加载 |
| 测试技巧 | 表驱动测试、mock 策略、集成测试模式 |
| 反模式与改进空间 | 代码中的妥协或可优化之处 |

### 通用质量要求

- 每个 Insight 必须锚定到项目的具体代码，不是抽象教科书建议
- Q&A 的 Evidence 必须引用具体 `file:line (FuncName)`，Inference 必须标明推断依据
- 引用代码时标注 `file:line (FuncName)`，方便跳转验证且重构后仍可定位
- 图表使用 ASCII 文本格式，保证可版本控制
- Code Insights 描述项目中的技术选择，不等同于推荐。`局限性`字段不得为空。

## 验证标准

### 事实正确性
- 每个 `file:line (FuncName)` 引用必须可跳转验证
- 抽样 3-5 个结论，核对是否与源码一致
- 所有图表中的模块/函数名必须与代码匹配
- 不存在的函数或类型不会被引用

### 内容质量
- 文档自包含：不需要额外交互即可理解
- 图表丰富：每层都有对应的可视化
- Q&A 有深度：Evidence 和 Inference 清晰分离
- Code Insights 实用：每个技巧锚定具体代码
- 概念有 example：每个概念都有场景说明 + 结构属性示例

### 可维护性
- 文档头部有完整元数据（含 skill_version、commit、时间、freshness scope）
- 引用带符号名，轻微重构不失效
- 后续补充不破坏已有结构

## 适用性调整

| 项目类型 | 调整 |
|----------|------|
| 多入口应用（如 Hydra） | L1 须枚举所有独立入口路径，L6 按入口分别画时序图 |
| 框架/库（如 React） | Layer 1 改为"使用者的心智模型是什么" |
| 过程式（如编译器） | Layer 2 改为"数据在每个阶段的形态" |
| 遗留代码 | Layer 4 切换为考古模式：git log 找拆分 commit |

## 生成流程

生成顺序：00-global.md 必须先完成，模块文档之间可并行。

### 默认模式（半自动）
00-global.md 生成 → 暂停等用户确认 → 模块文档并行生成
用户确认点：概念定义、模块分级（核心/重要/辅助）、入口路径枚举

### --auto 模式（全自动）
00-global.md 生成 → 模块文档并行生成（无暂停）

## 增量进化机制（两阶段）

### 进化触发器

每次用户提问后，AI 检查以下触发条件：

| # | 触发器 | 条件 |
|---|--------|------|
| T1 | 覆盖缺口 | 用户的问题在六层中找不到归属 |
| T2 | 深度不足 | 用户对某层追问（说明初始输出太浅） |
| T3 | 格式缺失 | 用户说"希望有 X 类型的图/表" |
| T4 | 跨项目模式 | 不同项目分析中反复出现相同 insight |
| T5 | 无效内容 | 用户明确表示某部分无价值 |

### 两阶段进化流程

**阶段一：记录候选改动**

命中触发器后，将改动记录到本文件末尾的"候选改动"区，**不修改正文**。

```
用户提问
  │
  ▼
AI 回答问题 + 更新分析文档
  │
  ▼
AI 检查 5 个触发器
  │
  ├── 未命中 → 结束
  │
  └── 命中 → 记录到"候选改动"区（不修改 Skill 正文）
```

**阶段二：确认后写回**

仅当以下条件之一满足时，才将候选改动写回 Skill 正文：
- **(a)** 跨两个以上项目重复出现同一候选改动
- **(b)** 用户明确确认该改动应写回
